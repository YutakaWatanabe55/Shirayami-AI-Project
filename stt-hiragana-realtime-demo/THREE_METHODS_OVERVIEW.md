# 3つのひらがな化方式（このデモでの実装）

このデモでは、まず音声からリアルタイム文字起こし（STT）を行い、その後に以下3方式を並行で走らせて比較しています。

## 2.1 Whisper文字種制限（`suppress_tokens`）

### 何をしているか
- Whisperのトークナイザを見て、かな系以外のトークンIDを抽出
- 推論時に `suppress_tokens` に渡して、漢字・英数字などの出力確率を抑制
- 結果をカタカナ→ひらがなへ正規化して表示

### 特徴
- デコード段階で「かな寄せ」する方式
- 漢字固定を避けやすい
- 読み崩れ（不自然な表記）が出ることがある

## 2.2 ヒント付き再推論 + 読み比較（ふりがなWhisper風 PoC）

### 何をしているか
- STT結果（またはユーザー指定ヒント）を `initial_prompt` としてWhisper再推論
- 再推論テキストを `cutlet`（fugashi/MeCab系）でひらがな化
- 2.1結果との差を編集距離で算出して指標化

### 特徴
- 文脈ヒントを使って読みを安定させる狙い
- このデモでは研究実装の完全再現ではなく、実運用向けPoC構成

## 3.1 後処理ひらがな化（ローカルNLP変換）

### 何をしているか
- 通常STT（漢字かな交じり）を先に取得
- その文字列を `cutlet` でひらがなへ変換

### 特徴
- 実装が比較的シンプル
- 変換速度が速い
- ただし、元のSTTが誤認識するとその影響は残る

## このデモのリアルタイム処理順

1. 録音チャンク（累積音声）をサーバーへ送信
2. リアルタイムSTTを更新
3. 2.1 / 2.2 / 3.1 を同時実行
4. 各モードの `latest` と `avg` 処理時間を表示

## 補足

- 実装ファイル:
  - `stt-hiragana-demo/scripts/run_realtime_worker.py`
  - `stt-hiragana-demo/server.js`
- 3方式は「同じ音声チャンク」を対象に比較しているため、速度差を見やすくしています。
